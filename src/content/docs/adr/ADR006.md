# ADR 006: Resilient Media Upload via Chunked Architecture

## Context
The Zamunda Digital Renaissance (ZDR) platform allows local businesses to upload media files (images, videos) to their profiles within a strict storage quota. However, a significant portion of the one million potential users reside in rural areas characterized by slow, unstable, and frequently disconnecting mobile network infrastructure. The system must guarantee that these users can successfully upload large files without experiencing constant failures or having to restart the process from scratch.

## Facing the Need for
* **Network Fault Tolerance:** The system must handle scenarios where a connection drops up to 10 times during a single upload session, allowing it to resume automatically once connectivity is restored.
* **Resource and Quota Control:** Uploads must be strictly validated against the business's allocated storage quota to prevent the abuse of state-funded infrastructure.
* **Server-Side Efficiency:** Preventing partially uploaded, abandoned files from permanently hoarding disk space on the central government infrastructure.
* **User Experience:** Eliminating the frustration of data waste and lost progress for users with limited or metered data plans.

## We Decided for
The implementation of a **Resilient Chunked Upload Architecture** utilizing a protocol designed for resumable file uploads (such as the TUS protocol or a custom chunked REST API implementation):
1. **Client-Side File Chunking:** The web-based Administration Portal will split files into small, manageable data blocks (chunks) before transmission.
2. **Stateful Upload Session Tracking:** The API gateway will issue a unique session token for each upload, tracking exactly how many bytes/chunks have been successfully written to the temporary storage.
3. **Byte-Range Resuming:** If a network disconnection occurs, the client will query the server for the last successfully received byte range and resume transmission from that precise offset, avoiding re-uploading previous data.
4. **Automated Lifecycle Garbage Collection:** A background worker will clean up incomplete upload sessions and temporary chunks that have been abandoned for more than 24 hours to enforce optimal resource usage.

## Neglecting
* **Standard Monolithic HTTP POST/PUT Uploads:** Reverted because standard multi-part uploads require a single, continuous connection. Any network drop forces the user to restart the entire upload, failing the core network resilience metric.
* **Client-Side Heavy Compression:** While compressing videos locally saves bandwidth, it introduces severe client-side device performance bottlenecks, which is unacceptable given the highly heterogeneous digital maturity and hardware variations among the users.

## Achieving
* **High Infrastructure Resilience:** Meets the strict requirement where an upload can recover seamlessly even after 10 consecutive network drops.
* **Bandwidth and Cost Optimization:** Minimizes redundant data transmission for rural users on unstable networks.
* **Quota Enforcement Alignment:** The system can validate file metadata and quota limits at the initialization phase before the actual data chunks are fully transmitted, saving processing power.

## Accepting That (We Will Have)
* **Increased System Complexity:** Requires maintaining upload state/metadata between the client and server during the upload lifecycle.
* **Temporary Storage Overhead:** The server must allocate temporary buffer zones to store incomplete chunks before they are assembled into the final object storage backend.
* **Slight Protocol Overhead:** Breaking files into chunks introduces minimal metadata overhead for individual HTTP requests, though this is negligible compared to the benefits of session resumability.
