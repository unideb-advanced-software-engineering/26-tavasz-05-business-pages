# ADR 005: Adoption of C4 Model for Architecture Documentation

## Context
The Zamunda Digital Renaissance (ZDR) program requires comprehensive and high-quality technical documentation due to its status as a state-funded initiative. The platform serves approximately one million local businesses with highly diverse digital literacy, while integrating with multiple external government and public APIs (Zamunda One, GitHub, National DNS). The engineering team needs a standardized way to communicate the system architecture to different stakeholders, including business analysts, government auditors, and software developers.

## Facing the Need for
* **Multi-level Abstraction:** The need to explain the system to non-technical government officials (User Classes) as well as technical implementers (Developers using the REST API).
* **Standardized Documentation:** As a state program, rigorous and high-quality project documentation is mandatory to ensure maintainability and transparency.
* **Complex Integration Mapping:** Clearly illustrating how the internal components (Domain Manager, Storage System) interact with external interfaces like the Zamunda One API, GitHub Webhooks, and the central DNS handler.
* **Elimination of Ambiguity:** Traditional, unstructured software architecture diagrams often lack clear boundaries and standardized semantics, leading to misinterpretation.

## We Decided for
The formal adoption of the **C4 Model** (Context, Containers, Components, and Code) as the core architectural blueprint standard for the Business Pages platform. Specifically, we will implement and maintain the first three levels of the framework:
1. **Level 1: System Context Diagram:** To define the boundaries of the ZDR platform, illustrating how Businesses, Public Visitors, and Administrators interact with the system alongside external state APIs.
2. **Level 2: Container Diagram:** To map the high-level technical architecture, explicitly separating the web-based Administration Portal, the Public Website frontend, the central API backend, the Message Broker for the event-driven architecture, and the Object Storage system.
3. **Level 3: Component Diagram:** To detail the internal structure of specific containers (e.g., decomposing the API backend into the Domain Manager, Content Sync Module, and Authentication Handler).

The diagrams will be maintained using a text-based, declarative tool (LikeC4) to ensure version control alignment within the repository.

## Neglecting
* **Ad-hoc UML/Unstructured Diagrams:** Reverted because custom shapes and lines without strict hierarchical meaning confuse stakeholders and fail to meet government-grade documentation standards.
* **Maintaining Level 4 (Code) Diagrams:** We rejected generating automatic or manual class/code-level diagrams, as they change too rapidly and add unnecessary maintenance overhead for the current team size.

## Achieving
* **Stakeholder Alignment:** Non-technical managers can easily understand the System Context (Level 1), while developers can immediately work with the Container and Component levels.
* **Rigorous Documentation Standards:** Fully satisfies the state requirement for clear, comprehensive, and auditable technical documentation.
* **Precise Boundary Definitions:** The relationship and responsibilities between internal services and external entities (like the Zamunda One authentication portal) are explicitly mapped out.
* **Version-Controlled Architecture:** Using text-based modeling (LikeC4) allows architectural diagrams to be reviewed, modified, and committed through Git alongside the source code.

## Accepting That (We Will Have)
* **Tooling Learning Curve:** The team must learn the syntax of the declarative modeling language (LikeC4) and ensure its pipeline integrations remain functional.
* **Maintenance Overhead:** Any major structural refactoring in the source code or functional requirements requires a manual or pipeline-triggered update to the C4 diagrams to prevent documentation rot.
